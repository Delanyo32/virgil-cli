## What to build

Populate two derived-fact relations at build time so queries don't have
to re-scan source on every invocation:

- `file_classification {path => is_test, is_barrel, is_generated}` — fed
  from `is_test_file` / `is_barrel_file` plus a small `is_generated`
  marker scan over the first 20 lines of each file.
- `nolint {file_path, line => suppressed_pattern}` — extracted from
  `// nolint:<pattern>` / `# nolint:<pattern>` / `/* nolint:<pattern>`
  comments in each file's source.

These join cleanly with `*symbol`, so queries like "complexity hotspots
excluding tests" stay one Cozoscript query instead of post-filtering
results in Rust.

## Triage Notes

> *This was generated by AI during triage.*

The original phase 4 stored seven `metric_*` relations
(cyclomatic / cognitive / nesting / length / comment-ratio /
afferent / efferent). After discussion the metrics were **deprecated
from the Cozo schema**:

- The existing `compute_metric` pipeline stage already computes metrics
  on demand via tree-sitter and the cost is negligible for the
  workspace sizes virgil-cli targets.
- Storing them duplicates source-of-truth and requires stale-fact
  invalidation work in issue 08 (incremental refresh) that buys nothing.
- `complexity_hotspots` (issue 05) is the only template that wanted a
  `*metric_*` JOIN; it's reframed as a Rust-side handler that calls
  `compute_metric` and merges with `*symbol` rows.

`file_classification` and `nolint` are kept because they *would* require
re-scanning every file's source/comments at query time. They are
genuinely materialised facts, not metric duplicates.
