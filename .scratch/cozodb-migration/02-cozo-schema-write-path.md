# 02 — Cozo dependency + schema + write path for cross-function graph

**Type:** AFK
**Label:** ready-for-agent

## Parent

`.planning/proposals/cozodb-migration.md` — Phase 2.

## What to build

Add `cozo` as a dependency and stand up the cross-function graph write
path. This is **additive** — the existing `CodeGraph` continues to be
built; Cozo relations are populated in parallel.

Use the **`cozo-ce`** fork (not upstream `cozo`) — see Triage Notes below
for the rationale. Required features at this stage: `storage-sqlite` +
`rayon`. `storage-rocksdb` is deferred to issue 07. `graph-algo` is
deferred to issue 10.

Scope:

- New `src/cozo/` module: `schema.rs`, `store.rs`, `writer.rs`, `queries.rs`.
  See proposal for line counts and responsibilities.
- All cross-function relations declared as `:create` in `schema.rs`:
  `file`, `symbol`, `callsite`, `call_arg`, `parameter`, `external_source`,
  `edge_defined_in`, `edge_calls`, `edge_imports`, `edge_exports`,
  `edge_contains`.
- All indices from the proposal's "Indices" section.
- `build_meta` + `build_meta_files` relations populated at end of build.
- The graph absorber writes Cozo rows in batched (~10k) transactions, fed
  from the existing parse-worker channel.
- Monotonic Int IDs assigned in the absorber.
- A `--cozo-only` test flag (or equivalent hidden CLI option) that runs a
  handful of trivial Cozoscript queries against the populated DB to prove
  the write path works.

The Cozo store starts in-memory for this phase — RocksDB persistence comes
in Phase 7 (issue 07).

## Triage Notes

> *This was generated by AI during triage.*

A scouting attempt to add `cozo` surfaced two upstream packaging issues
on the current Rust toolchain (rayon 1.11). Documenting here so the
implementing agent doesn't rediscover them:

1. **Use `cozo-ce`, not `cozo`.** The `cozo` crate (0.7.6) and `cozo-ce`
   (0.7.13-alpha.3) share most code, but the upstream `cozo` 0.7.6 has
   the rayon packaging bug below *and* no newer publish to land a fix.
   `cozo-ce` is the community-edition fork; same API surface.

2. **Do NOT enable `graph-algo` yet.** That feature transitively pulls
   `graph_builder 0.4.1`, an abandoned crate whose `into_par_iter()` call
   is incompatible with rayon ≥ 1.8. It will not compile. `graph-algo`
   is only needed by issue 10; that issue should investigate either a
   `[patch.crates-io]` fork of `graph_builder` or a workspace-local rayon
   pin at that point.

3. **You must enable the `rayon` feature explicitly.** Cozo's `lib.rs`
   calls `rayon::spawn` at module top level, but the `rayon` dep is
   feature-gated. Without `features = ["rayon", ...]` the crate fails
   to compile even with `graph-algo` disabled.

Working `Cargo.toml` line from scouting (confirmed compiling on this
machine, 413 lib tests still green):

```toml
cozo = { package = "cozo-ce", version = "0.7.13-alpha.3", \
         default-features = false, features = ["storage-sqlite", "rayon"] }
```

The scouting code (a minimal `src/cozo/{mod,store,schema,writer,queries}.rs`
that round-trips `symbol` + `edge_calls` rows through an in-memory store)
was reverted before commit; the implementing agent should reproduce it
from scratch following the proposal's module breakdown.
