# 09 — DEPRECATED: petgraph stays

**Type:** N/A
**Label:** wontfix

## Status

**Deprecated.** Issue 09 was scoped as a one-day cleanup with the
premise that issue 06 (The Deletion) had removed every consumer of
`petgraph::DiGraph`. After issues 03 and 06 were both narrowed, that
premise no longer holds.

## Current petgraph consumers (16 files)

```
src/graph/mod.rs                  CodeGraph stores DiGraph<NodeWeight, EdgeWeight>
src/graph/builder.rs              uses NodeIndex during absorption
src/graph/cfg.rs                  FunctionCfg uses DiGraph<BasicBlock, CfgEdge>
src/graph/resource.rs             walks CodeGraph + FunctionCfg
src/graph/taint/types.rs          taint types use NodeIndex
src/graph/taint/engine.rs         taint engine walks CodeGraph + CFGs

```

## Triage Notes

> *This was generated by AI during triage.*

Dropping petgraph today would require:

1. Rewriting `CodeGraph` to use a non-petgraph adjacency list. Mechanical
   but touches every node-iteration and edge-walking site.
2. Rewriting all 9 language CFG builders to emit into a non-petgraph
   structure (this is the bulk of the issue 03 scope that got
   deprecated).
3. Updating `taint::engine` and `resource::ResourceAnalyzer` to traverse
   the new structure.
4. Updating `cozo::from_code_graph::populate` to walk via the new API
   instead of `EdgeRef`.

That's ~16 files of mechanical translation plus the risk of subtle
graph-traversal regressions in taint and resource analyses. Same scope
as the issue 03 work we declined.

**The reason for issue 09 in the original plan** was that the migration
would eventually have *no* in-memory graph at all — everything would
live in Cozo facts. That outcome required issue 03 (CFG facts) and
issue 06's full taint/resource/cfg deletion, both of which we narrowed
or deprecated.

**Cost-benefit:** petgraph is a small (~150 KB), well-maintained
dependency. The Cargo.lock entry is invisible to end-users. The only
visible payoff of removing it is the proposal's "cargo tree | grep
petgraph returns nothing" check.

Recommendation: leave petgraph in. If a future motivation arises (e.g.
the in-memory graph itself is the bottleneck for cold-start time on
large workspaces), file a new issue scoped to "redesign CodeGraph" and
delete petgraph as a side effect.
