## What to build

Add the user-facing Cozoscript query surface to the `projects query`
subcommand and ship the first 10 built-in templates.

CLI surface (additive â€” existing `--q '{json}'` keeps working until
Phase 6 deletes it):

- `--cozoscript '<inline>'` â€” power-user mode
- `--file <path.cozoql>` â€” load script from file

- `--param k=v` â€” parameter binding (repeatable). Parameters bound as
  `$param` in Cozoscript; never interpolated into the script text

auto-formatted as audit findings. Other shapes use existing
`--format json|csv|table`.

## Triage Notes

> *This was generated by AI during triage.*

The proposal includes worked Cozoscript for only 3 of the 10 templates
(`find_callers`, `find_cycles`, `complexity_hotspots`). The other 7 must
be authored from scratch. Subtle Datalog errors can pass a small fixture
test but miss real findings, so each template needs a real-corpus smoke
test (counts of rows returned on a representative project), and the
results should be posted in the PR description for review.

**`complexity_hotspots` is not a pure Cozoscript template.** Issue 04
deprecated `metric_*` facts from the schema (see
`04-metric-facts.md`'s Triage Notes), so the JOIN against
`*metric_cyclomatic_complexity` in the proposal no longer works.
Implement `complexity_hotspots` as a Rust-side `--template` handler
that:

1. Runs `?[id, name, file_path, start_line, end_line] := *symbol{...}`
   with `*file_classification{path: file_path, is_test: false}` as a
   guard.
2. For each row, calls `crate::pipeline::stages::compute_metric::*` to
   compute cyclomatic + length on demand (tree-sitter is fast enough
   that this is a non-issue for typical workspaces).
3. Filters by the `cc_threshold` / `length_threshold` `--param` values
   and emits the audit-shaped `(file, line, severity, pattern, message)`
   tuple.

This is the only template that needs a Rust escape hatch â€” the rest are
pure Cozoscript.
